## 1. 分层架构概述

SCI数据仓库采用经典的五层架构设计，实现数据从源头到应用的全链路管理。各层职责分明，层层递进，确保数据质量与业务需求的平衡。

**分层架构图：**

```
┌─────────────────────────────────────────────────────────────┐
│                     数据源 (Source Systems)                   │
└─────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────┐
│  sci_idl  接口层 (Interface/Data Landing Layer)             │
│  接收原始数据，保持数据原貌                                  │
└─────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────┐
│  sci_ods  操作数据层 (Operational Data Store)               │
│  数据清洗标准化，跨源整合                                    │
└─────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────┐
│  sci_dw   数据仓库层 (Data Warehouse Layer)                 │
│  维度建模，主题域分析                                        │
└─────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────┐
│  sci_tmp  临时数据层 (Temporary Layer)                     │
│  中间计算，结果暂存                                          │
└─────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────┐
│  sci_ads  应用数据层 (Application Data Store Layer)          │
│  分析结果，报表服务                                          │
└─────────────────────────────────────────────────────────────┘
```

**独立应用层：**

```
┌─────────────────────────────────────────────────────────────┐
│  sci_app 应用系统层 (Application System Layer)               │
│  用户管理，菜单权限，系统配置                                  │
└─────────────────────────────────────────────────────────────┘
```

## 2. 分层职责说明

### sci_idl - 接口层 (Interface/Data Landing Layer)

**定位：** 数据进入数据仓库的第一站

**核心职责：**
- 接收并存储来自外部系统的原始数据
- 保持数据的原始形态，不进行过多清洗转换
- 为后续数据处理提供基础数据源
- 支持数据溯源和原始数据恢复

**设计原则：**
- 保留数据原始字段和格式
- 添加数据接入时间戳便于追踪
- 支持增量数据和全量数据同步
- 与源系统数据结构保持一致或高度相似

**当前状态：** 建设中，暂无预定义表结构

---

### sci_ods - 操作数据层 (Operational Data Store)

**定位：** 数据质量提升层，承上启下

**核心职责：**
- 对IDL层原始数据进行清洗和标准化处理
- 实现跨源数据的一致性整合
- 建立基础数据质量规则和校验机制
- 为DW层提供高质量的业务数据

**设计原则：**
- 遵循3NF范式进行数据建模
- 统一数据编码和格式标准
- 保留数据变更历史追踪能力
- 确保与业务系统数据同步

**当前状态：** 建设中，暂无预定义表结构

---

### sci_dw - 数据仓库层 (Data Warehouse Layer)

**定位：** 核心分析层，维度建模

**核心职责：**
- 对ODS层数据进行维度建模和主题域划分
- 构建星型/雪花模型的事实表和维度表
- 支持多维度业务报表和分析需求
- 提供历史数据快照和趋势分析能力

**设计原则：**
- 采用Kimball维度建模方法论
- 事实表遵循3NF规范，维度表可适度反规范化
- 支持缓慢变化维（SCD）处理数据历史变更
- 确保数据一致性和业务口径统一

**当前状态：** 已完成，表结构详见[表清单统计](#3-表清单统计)

---

### sci_tmp - 临时数据层 (Temporary Layer)

**定位：** 工作区域，中间计算

**核心职责：**
- 存储ETL过程中的中间计算结果
- 支持数据清洗和转换的临时存储
- 为复杂计算提供分步结果保存
- 支持数据验证和质量检查

**设计原则：**
- 采用扁平化数据结构提升计算效率
- 支持海量数据的高并发写入
- 临时表使用后及时清理释放空间
- 可根据业务需求动态创建和销毁表结构

**当前状态：** 建设中，暂无预定义表结构

---

### sci_ads - 应用数据层 (Application Data Store Layer)

**定位：** 数据服务层，面向应用

**核心职责：**
- 存储面向应用的数据分析结果
- 提供业务报表和仪表盘数据支持
- 实现KPI指标和度量的预计算
- 支持实时和准实时的数据服务

**设计原则：**
- 采用星型或雪花模型进行数据建模
- 优化查询性能支持快速报表生成
- 数据按业务主题域进行组织划分
- 支持历史数据分析对比需求

**当前状态：** 建设中，暂无预定义表结构

---

### sci_app - 应用系统层 (Application System Layer)

**定位：** 独立于数据仓库的应用系统表层

**核心职责：**
- 存储应用系统所需的用户管理信息
- 提供菜单结构和权限控制功能
- 管理系统配置和参数设置
- 支持审计日志和操作记录

**设计原则：**
- 支持灵活的菜单层级结构
- 实现基于角色的访问控制（RBAC）
- 记录关键业务操作的审计日志
- 与数据仓库分层隔离，单独维护

**当前状态：** 建设中，暂无预定义表结构

## 5. 快速开始

### 5.1 数据库初始化顺序

数据库初始化需按照以下顺序执行脚本，确保依赖关系正确：

```bash
# 1. 创建数据库Schema
psql $DATABASE_URL -f create_schema.sql

# 2. 设置角色权限
psql $DATABASE_URL -f set_permission.sql

# 3. 创建数据表结构（sci_dw层）
psql $DATABASE_URL -f sci_dw/dw_create_table.sql

# 4. 初始化种子数据（sci_dw层）
psql $DATABASE_URL -f sci_dw/seed_data.sql
```

### 5.2 初始化检查

执行完成后，可通过以下SQL验证：

```sql
-- 检查Schema是否创建成功
SELECT schema_name FROM information_schema.schemata WHERE schema_name LIKE 'sci_%';

-- 检查表是否创建成功
SELECT table_schema, table_name FROM information_schema.tables WHERE table_schema = 'sci_dw';

-- 检查数据是否插入成功
SELECT COUNT(*) as table_count FROM sci_dw.factory_master;
```

### 5.3 常见问题

| 问题 | 解决方案 |
| ---- | -------- |
| 权限不足 | 检查是否已执行 set_permission.sql |
| 表已存在 | 可使用 DROP TABLE 删除后重新创建，或添加 IF NOT EXISTS |
| 数据插入失败 | 检查外键约束是否满足 |

## 6. 账号权限说明

数据库角色分为两类：a_appconnect（开发）、application（应用服务）。

### 6.1 角色说明

| 角色 | 用途 | 密码配置 |
| ---- | ---- | -------- |
| a_appconnect | 数据库开发维护，完全访问 | developer_password |
| a_application | 应用运行时连接账户 | app_password |

**注意：** 生产环境请务必修改默认密码。

### 6.2 Schema 权限矩阵

| Schema | a_appconnect | a_application |
| ------ | ------------- | ---------------- |
| SCI_IDL | ALL PRIVILEGES | SELECT, INSERT, UPDATE, DELETE |
| SCI_ODS | ALL PRIVILEGES | SELECT, INSERT, UPDATE, DELETE |
| SCI_DW | ALL PRIVILEGES | SELECT, INSERT, UPDATE, DELETE |
| SCI_TMP | ALL PRIVILEGES | SELECT, INSERT, UPDATE, DELETE |
| SCI_ADS | ALL PRIVILEGES | SELECT, INSERT, UPDATE, DELETE |
| **SCI_APP** | ALL PRIVILEGES | **ALL PRIVILEGES** |

### 6.3 权限说明

| 权限类型 | 说明 |
| -------- | ---- |
| USAGE | 允许使用 schema |
| SELECT | 读取数据 |
| INSERT | 插入数据 |
| UPDATE | 更新数据 |
| DELETE | 删除数据 |
| ALL PRIVILEGES | 完全控制（包括 DDL） |

### 6.4 SCI_APP 特殊说明

SCI_APP schema 存放应用系统表（用户、菜单、角色等），应用服务账户（a_application）对其拥有完全控制权限，以支持应用运行时动态管理用户权限和系统配置。
